# Indonesia Tourism Ontology (Bali · NTB · NTT)

An OWL 2 DL ontology covering tourism in three Indonesian provinces — **Bali**,
**West Nusa Tenggara (NTB)**, and **East Nusa Tenggara (NTT)** — populated
automatically from DBpedia and verified for consistency with the **HermiT**
reasoner.

The TBox (classes, properties, axioms) is hand-modeled in Protégé. The ABox
(individuals and their relations) is generated from DBpedia SPARQL queries via
a small modular Python pipeline.

## Project structure

```
.
├── main.py             # Orchestrator: populate -> enrich -> save -> reason
├── config.py           # Constants, IRIs, file paths, domain mappings (data, no logic)
├── graph_utils.py      # Tiny RDF helpers (slugify, add_individual, …)
├── populate.py         # DBpedia SPARQL queries -> individuals (rdf:type)
├── enrich.py           # Relational triples (locatedIn, hasActivity, …)
├── reasoning.py        # HermiT consistency check via owlready2
├── visualize.py        # Interactive HTML graph (PyVis)
├── ontology/
│   ├── schema.owl          # TBox: classes, properties, axioms
│   ├── data.owl            # ABox: individuals (regenerated by main.py)
│   ├── travel.owl          # Imported Stanford travel ontology
│   └── catalog-v001.xml    # Protégé import catalog
├── requirements.txt
└── README.md
```

## Setup

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

You also need a working **Java** runtime on `PATH` (HermiT is a `.jar`).

## Usage

```bash
# Full pipeline: query DBpedia -> enrich -> save data.owl -> run HermiT
python main.py

# Render data.owl + schema.owl as an interactive HTML graph
python visualize.py
open ontology_graph.html
```

`main.py` is idempotent — it wipes `ontology/data.owl` to a clean skeleton on
each run before re-populating.

## Pipeline

| Step | Module | What it does |
|---|---|---|
| 1 | `main.py` | Reset `data.owl` to blank skeleton |
| 2 | `populate.py` | Query DBpedia, add `rdf:type` triples for each class |
| 3 | `enrich.py` | Add relational triples (locatedIn, hasActivity, hasTouristAttraction, …) |
| 4 | `main.py` | Save the merged graph to `ontology/data.owl` |
| 5 | `reasoning.py` | Load with owlready2, run HermiT, report class extensions |

Typical numbers per run:

- ~250 triples after populate (rdf:type + minimal locatedIn)
- ~440 triples after enrich (4× more relational density)
- HermiT verdict: **CONSISTENT ✓**

## Ontology classes

`Province`, `Country`, `Island`, `City`, `Beach`, `Park`, `Volcano`, `Museum`,
`Temple`, `Festival`, `ReligiousCeremony`, `Hotel`, `Activities`,
`TouristAttraction` (and subclasses).

## Object properties used

- `locatedIn` (parent), `locatedInCountry`, `locatedInProvince`, `locatedInIsland`
- `hasTouristAttraction` (City -> TouristAttraction)
- `hasActivity` (TouristAttraction -> Activities)

## Data sources

All ABox individuals come from **DBpedia** (`https://dbpedia.org/sparql`).
DBpedia categorisation is uneven across the three provinces, so each class
uses a slightly different query strategy — see comments in `populate.py`.
For NTB and NTT, some categories don't exist (e.g., `Beaches_of_Lombok`); the
queries fall back to category intersections like
`Beaches_of_Indonesia ∩ Landforms_of_Lombok`.

## Notes & known limitations

- DBpedia has very sparse data for NTB ceremonies/festivals. `Bau_Nyale` and
  `Perang_Topat` (the famous Sasak rituals) don't exist as DBpedia resources.
- `Pasola` and `Caci` (NTT) are categorised as martial arts in DBpedia even
  though they are religious rituals; they're pulled explicitly via `VALUES`.
- `hasAccommodation` is `owl:equivalentProperty` to `travel.owl#hasAccommodation`
  whose domain (`travel:Destination`) clashes with our `City`. The enrichment
  step skips it to keep the ontology consistent.
- `originatesFrom` has domain `Food` in the schema — it's not used for
  ceremonies (would force `ReligiousCeremony ⊑ Food`).
